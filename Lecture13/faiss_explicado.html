Las tres fases de search().

1
Preparar el vector de consulta
Convertir a float32 y normalizar: q /= np.linalg.norm(q) — la misma normalización que en la inserción. Ambos lados deben vivir en la esfera unitaria; si no, producto interno ≠ similitud coseno. Se hace reshape(1, dim) porque FAISS espera un batch.
2
Búsqueda FAISS — espacio vectorial puro
scores, indices = self.index.search(q.reshape(1,-1), k)
Devuelve dos arrays de forma (1, k). FAISS no conoce tus documentos.

El guard idx >= 0 maneja el padding cuando hay menos de k resultados (FAISS rellena con -1).
3
Puente: entero → DocumentChunk
self.documents[idx] recupera el objeto completo. id_to_index no se usa aquí — la búsqueda va en dirección entero FAISS → documento. id_to_index es la dirección inversa: id del doc → entero (lookups, dedup, borrado).
42 → documents[42] → (DocumentChunk, 0.91)

Todo el sistema depende de un invariante estructural: la posición i en FAISS siempre corresponde a self.documents[i].

Por qué se mantiene
En add(), pos = len(self.documents) se captura antes del append. Luego self.documents.append(doc) y self.index.add(vec) ocurren en la misma iteración del loop, en el mismo orden. Por eso la n-ésima llamada a index.add() siempre inserta el mismo doc que quedó en la posición n de self.documents.
Por qué puede romperse
faiss.IndexFlatIP no tiene operación de borrado. Si eliminas un documento de self.documents sin reconstruir el índice FAISS completo, las posiciones se desplazan y el mapeo queda inválido. Ésa es una de las razones por las que existen las bases de datos vectoriales de producción (Qdrant, Weaviate): gestionan las mutaciones de forma nativa.
Dirección de uso: resumen
search() entero FAISS → self.documents[i] usa documents
get_by_id() doc_id str → int → documents[i] usa id_to_index
dedup check doc_id in id_to_index → verificación O(1) usa id_to_index
